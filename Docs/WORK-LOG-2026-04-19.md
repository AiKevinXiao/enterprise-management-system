## 一、文件操作类问题

### 1. BOM 导致的编码问题

**现象**：前端 `login.html` 打开一片空白，服务器响应头已设置 `charset=utf-8`。

**根因**：`login.html` 文件开头有 UTF-8 BOM（`EF BB BF`），浏览器将 BOM 当作正文解析，导致 `<!DOCTYPE` 等标签被截断，HTML 结构错乱。

**解决**：用十六进制编辑器或 PowerShell 去除 BOM。

```powershell
# 读取文件，去掉开头 3 字节的 BOM，剩余内容写回
$bytes = [System.IO.File]::ReadAllBytes("login.html")
[System.IO.File]::WriteAllBytes("login.html", $bytes[3..($bytes.Length-1)])
```

**预防**：新建文本文件时选择"UTF-8 无 BOM"编码；VS Code 右下角确认编码格式。

---

### 2. 行尾符 `\r\r\n` 混入导致文件损坏

**现象**：`user-management.html` 部分行以 `\r\r\n` 结尾（连续两个回车），导致：
- 直接在 PowerShell 中 `echo` 或 `Set-Content` 追加内容时代码被截断
- `edit` 工具匹配旧字符串时完全失败（`oldText` 里没有 `\r`）
- 文件被意外清空为 0 字节（PowerShell 写文件遇到非法行尾时行为异常）

**排查过程**：
1. 先以为是文件被锁定，反复尝试写入 → 文件被清空
2. 用十六进制查看器确认是 `\r\r\n` 问题
3. 用 Python 脚本修复：按 `latin-1` 读 → UTF-8 解码 → 替换 → UTF-8 写回

```python
with open('user-management.html', 'rb') as f:
    raw = f.read()
text = raw.decode('latin-1').replace('\r\r\n', '\r\n').replace('\r\n', '\n')
with open('user-management.html', 'w', encoding='utf-8', newline='\n') as f:
    f.write(text)
```

**教训**：
- Windows 换行符混用是常见问题，跨工具操作文件（Git / VS Code / PowerShell / Python）时尤其容易出现
- **首选 VS Code 直接编辑**，不要在 PowerShell 中 echo 大段内容
- edit 工具匹配失败时，优先怀疑行尾符不一致
- 对来源不明的文件，先 `Get-Content -AsByteStream` 确认字节结构再操作

### 3. PowerShell 5.x 不支持 `&&` 链式语法

**现象**：PowerShell 5.x 中执行 `git add -A && git commit -m "msg"` 报错。

**根因**：`&&` 是 PowerShell 7.0+ 的特性，Windows PowerShell 5.x（`powershell.exe`）不支持。

**解决**：分开执行，或用 `;` 替代。

```powershell
# 方案一：分开执行
git add -A
git commit -m "fix: something"

# 方案二：用 ; 分隔（可替代 &&，串行执行）
git add -A ; git commit -m "fix: something"
```

**补充**：检查当前 PowerShell 版本：
```powershell
$PSVersionTable.PSVersion
```

---

### 4. `edit` 工具部分替换未生效

**现象**：执行 `edit` 替换后，文件内容未变化，控制台无报错。

**根因**：目标字符串中包含行尾符或不可见字符，`oldText` 里的换行符格式与实际文件不符。

**解决**：
1. 用 `Get-Content` 读取文件内容并显示十六进制辅助信息
2. 或用 Python 脚本直接按字节操作（见上文）

**教训**：`edit` 工具适合纯文本匹配替换；涉及行尾符、混合编码文件时，用 Python 脚本更可靠。

## 二、服务与进程类问题

### 1. 后端改了代码但没重启服务（今日最费时）

**现象**：修改了 `backend/routes/roles.js` 添加 `permission_codes` 支持，前端测试时接口返回 200 但数据库无变化，前后端均无报错。

**根因**：Node.js 服务启动后代码加载到内存，修改文件不会自动重载服务。

**排查过程**：
1. ✅ 检查浏览器 Network 面板 → 请求已发出，body 正确
2. ✅ 前后端加 console.log 追踪 → 数据格式正确
3. ❌ **忽略了一个关键事实**：后端进程启动时间（13:50）早于文件修改时间（14:02）

**快速定位方法**：
```powershell
# 进程启动时间
(Get-Process -Id <PID>).StartTime

# 文件修改时间
(Get-Item "backend/routes/roles.js").LastWriteTime

# 对比：如果进程启动时间 < 文件修改时间 → 服务在跑旧代码
```

**教训**：
- **修改后端代码后，必须重启服务**
- 排查"接口返回 200 但数据没变"时，**第一步就检查进程时间 vs 文件时间**，不要在代码逻辑里绕太久
- 养成习惯：改完代码 → 重启服务 → 测试，不要跳过

---

### 2. 前端改了代码但没刷新页面

**现象**：前端代码已修改，但页面行为没有变化。

**根因**：浏览器缓存了旧的 HTML / CSS / JS 文件。

**解决**：
- 强制刷新：`Ctrl + Shift + R`（Windows）
- 或开发者工具 → Network → 勾选 "Disable cache" → 刷新
- 服务器已设置 no-cache 响应头，如仍缓存检查浏览器缓存配置

**教训**：前端调试时建议开发者工具 Network 面板勾选 "Disable cache"，避免缓存干扰。

## 三、代码编写类问题

### 1. 变量初始值用 truthy 字符串导致条件判断失效

**现象**：点击左侧角色列表，右侧权限面板不更新。

**根因**：
```javascript
let currentRoleId = 'admin';  // 'admin' 是 truthy

// 后续逻辑：
if (currentRoleId) {           // 'admin' 为 true，此条件永远成立
    currentRoleId = roles[0].id;  // 这行永远不会执行
}
```
结果：`currentRoleId` 始终是字符串 `'admin'`，API 调用 404。

**教训**：
- 表示"未设置"的状态，初始化为 `undefined` 或 `null`
- JS falsy 值：`undefined`、`null`、`0`、`''`、`false`（适合作为"空"状态）
- JS truthy 值：所有非空字符串（包括 `'0'`、`'false'`、`'admin'`）——不适合做占位

---

### 2. API 字段名不一致导致静默失败

**现象**：前端发送 `{ permission_codes: [...] }`，后端期望 `{ permission_ids: [...] }`。接口返回 200，数据库无变化，无任何报错。

**根因**：
```javascript
// 后端代码（解构 permission_ids，permission_codes 被忽略）
const { permission_ids } = req.body;  // undefined
if (permission_ids && permission_ids.length) {  // 跳过
    // 更新逻辑
}
```

**教训**：
- 前后端字段命名必须完全一致，**不要有同义不同名的情况**
- 后端接收请求时，显式打印日志确认字段：`console.log('req.body:', req.body)`
- 静默失败是最贵的 bug：响应正常但逻辑跳过，排查极费时间
